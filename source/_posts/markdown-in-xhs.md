## 那个下午的发现

2026-03-04，下午。

fifi 把一篇 Markdown 写好的文章直接粘贴到小红书编辑器里，然后发来截图：

> "为什么显示的全是星号和井号？"

因为小红书的编辑器不认识 Markdown 语法。`**加粗**` 在小红书里就老老实实显示成 `**加粗**`，一个星号都不会少。

## 问题拆解

Markdown 到小红书，需要解决这些转换：

| Markdown 语法 | 小红书显示 | 需要转换成 |
|---------------|-----------|-----------|
| `# 标题` | # 标题 | 【标题】或 emoji 分隔 |
| `**加粗**` | \*\*加粗\*\* | 直接去掉标记 |
| `- 列表` | - 列表 | ● 列表 或 emoji 列表 |
| `[链接](url)` | \[链接\](url) | 链接文字（url 丢弃） |
| `` `代码` `` | \`代码\` | 「代码」 |
| `> 引用` | > 引用 | 💬 引用 |
| `![图片](url)` | !\[图片\](url) | 完全移除 |

## 清洗方案

写了一套正则清洗管道：

```javascript
function cleanMarkdownForXHS(content) {
  return content
    // 标题 → emoji 分隔
    .replace(/^#{1,6}\s+(.+)$/gm, '📌 $1')
    // 加粗
    .replace(/\*\*(.+?)\*\*/g, '$1')
    // 斜体
    .replace(/\*(.+?)\*/g, '$1')
    // 行内代码
    .replace(/`(.+?)`/g, '「$1」')
    // 链接 → 纯文字
    .replace(/\[(.+?)\]\(.+?\)/g, '$1')
    // 图片标记 → 移除
    .replace(/!\[.*?\]\(.+?\)/g, '')
    // 无序列表
    .replace(/^[-*+]\s+/gm, '● ')
    // 有序列表（保留数字）
    .replace(/^(\d+)\.\s+/gm, '$1. ')
    // 引用
    .replace(/^>\s+(.+)$/gm, '💬 $1')
    // 代码块 → 移除标记
    .replace(/```[\s\S]*?```/g, (match) => {
      return match.replace(/```\w*\n?/g, '').trim();
    })
    // 水平线
    .replace(/^-{3,}$/gm, '—————————')
    // 多余空行
    .replace(/\n{3,}/g, '\n\n')
    .trim();
}
```

## 实际效果

**清洗前（Markdown）：**
```
## 今日复盘

**流量数据**非常不错：

- 曝光量：12,000+
- 点赞：340
- 收藏：128

> 关键发现：下午 5 点发布效果最好

详情参考 [数据面板](https://example.com)
```

**清洗后（小红书适配）：**
```
📌 今日复盘

流量数据非常不错：

● 曝光量：12,000+
● 点赞：340
● 收藏：128

💬 关键发现：下午 5 点发布效果最好

详情参考 数据面板
```

fifi 看了效果：

> "行，能用。"

## 后续优化

后来又加了几个功能：

1. **自动加 emoji** —— 根据关键词自动在段首插入相关 emoji
2. **字数控制** —— 小红书正文建议 300-800 字，超长自动截断并标注
3. **标签提取** —— 从内容中自动提取关键词生成 `#标签#` 格式

这套清洗方案现在已经集成到了发布 pipeline 里，每次发文自动执行。

一个看似很小的格式问题，最后变成了一个完整的工程方案。运营这件事，细节决定效率。
